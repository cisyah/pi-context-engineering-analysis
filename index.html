从 Pi-Agent 看上下文工程—— 生产级 Agent 的策略拆解

总览

overview

从 AI 产品经理视角，把 pi-agent 的上下文机制拆成产品目标、用户问题、机制选择、取舍和失效后果。

“上下文不是 prompt 拼接，而是一条持续重建的产品管道。”

核心判断：pi-agent 的价值在于把模型输入变成可观测、可恢复、可裁剪的系统。

这是什么

将 pi-agent 当作生产级 AI Agent 产品系统来拆解：上下文从哪里来、如何进入模型、如何被过滤、压缩、恢复，以及失控时系统如何止损。

为什么要拆解 Pi-Agent 项目

  • 它把 prompt 从静态文案升级成可维护系统：资源发现、设置合并、技能索引、会话历史和工具定义共同参与上下文构建。
  • 它暴露了生产级 Agent 的真实矛盾：模型既不能缺上下文，也不能拿到错误上下文，更不能被无关内容淹没。
  • 它的任务生命周期完整覆盖启动、会话恢复、agent loop、工具结果、compaction、分支和韧性恢复，适合观察上下文如何长期流动。
  • 它能给 AI 产品经理一个可复用判断框架：什么信息应稳定预设，什么信息应由用户提供，什么信息必须由系统动态裁剪。

项目概况

维度项目概况产品含义
项目类型面向编码任务的 AI Agent / CLI。
  • 不是单轮问答。
  • 是持续任务执行系统。
核心对象AgentContext、会话树、工具、Entry、CompactionEntry。
  • 围绕“下一轮模型应该看见什么”组织。
  • 把历史、工具和状态放进同一条上下文管道。
上下文入口buildSessionContext()、transformContext()、convertToLlm()。
  • 从会话历史到模型消息。
  • 每轮调用前持续重建和过滤。
风险约束防欠给、防错给、防过给。
  • 上下文质量比数量更关键。
  • 预算、来源和时机都要被产品化管理。
运行状态activeToolNames、steering、follow-up、分支摘要、模型热替换。
  • 用户干预要可恢复。
  • 系统状态要能影响下一轮上下文。

核心洞察

  1. 上下文不是 prompt 拼接。它是一条从资源、历史、工具到模型消息的产品管道，每个边界都在重新决定“给什么”。
  2. 压缩不是删历史。好的 compaction 是把早期工作重写成可恢复摘要，并保护 user → assistant → toolResult 的结构完整性。
  3. 工具暴露就是产品权限。activeToolNames 不只是省 token，它决定模型此刻被允许做什么，也决定它会被什么诱导。
  4. 稳定性来自来源分层。开发者预设、用户提供、LLM 自建三层来源要分开治理，否则偏差会在会话中自我强化。

01

总览：上下文工程不是 prompt 拼接

从 AI 产品经理视角看，pi 的核心命题可以压缩成一句话：把正确的信息，在正确的时间，以正确的形式，喂给正确的大模型。

这句话里的每个词都对应一个产品风险。信息不正确，模型会沿着错误前提推理；时间不正确，模型会在不该写文件的时候写文件；形式不正确，模型看到了内容却读不出结构；模型不正确，简单任务浪费成本，复杂任务又不够稳。

策略：pi 没有把上下文当成一次性字符串，而是把它设计成一条持续流动的管道。启动、会话恢复、用户插话、工具执行、压缩、分支切换，每个环节都可能改写下一次模型调用看到的内容。

三个基础矛盾

上下文工程本质上是在管理三个互相拉扯的约束：欠给、错给、过给。它们不是孤立问题，而是会互相触发的产品链路问题。

上下文工程的三类失败Tradeoff

  ROOT["上下文工程的三类失败"]
  ROOT --> UNDER["欠给

  ROOT --> WRONG["错给

  ROOT --> OVER["过给
信息太多，注意力稀释或上下文溢出"]
  UNDER --> UFIX["产品解法

  WRONG --> WFIX["产品解法

  class ROOT section-root
  class UNDER,UFIX section-1
  class WRONG,WFIX section-2
  class OVER,OFIX section-3
问题原因产品后果
欠给缺少编码规范、工作目录、历史决策或工具能力说明。模型开始脑补 API，忽略用户约束，产出看似合理但无法落地。
错给工具定义和实际能力不匹配，文件内容过时，路径或状态恢复错误。模型在错误前提上做正确推理，结果全错，排查成本高。
过给把完整历史、全部工具、所有技能一次性塞入上下文。上下文溢出、注意力稀释、延迟增加，关键指令被噪音淹没。

最终收敛为三个入口

无论上游有多少资源文件、设置、技能、会话 Entry、压缩摘要，进入 agent loop 的上下文最终都收敛为 AgentContext 的三个要素。

要素说明产品含义
systemPrompt角色定义、工具指南、技能索引、项目上下文、日期和 cwd。决定模型的行为边界和基础世界观。
messages用户消息、助手响应、工具调用和结果、压缩摘要。承载任务历史和当前工作状态。
tools当前暴露给模型的工具集合，可被 activeToolNames 过滤。决定模型此刻能做什么，也决定它会被什么诱导。
02

生命周期：一次任务中的上下文流动

pi 的上下文不是在启动时构建一次，然后原封不动地传给模型。它会随着任务推进不断重建、过滤、注入和压缩。

策略：把生命周期拆成启动准备、循环运行、收尾维护三组。启动准备决定初始骨架，循环运行处理高频变化，收尾维护负责压缩和持久化。

解决的问题：编码任务不是一次问答，而是持续决策过程。用户可能插话，工具可能失败，模型可能换档，历史可能压缩。只有把上下文做成流水线，系统才有机会在每个边界修正方向。

不这样做会怎样：上下文只能线性膨胀。系统要么把所有历史硬塞给模型，要么粗暴截断早期信息；用户中途干预也只能变成一条普通消息，模型无法理解它是“立即转向”还是“之后再做”。

03

启动：资源发现和信任边界

启动阶段的产品目标是建立稳定基线：加载该加载的规范，拒绝不该信任的项目资源，并把全局与项目级设置合并成确定性的上下文输入。

分层资源发现

全局资源 (agentDir)
  ├── AGENTS.md / CLAUDE.md
  ├── SKILL.md
  ├── PROMPT.md
  └── settings.json
          ↓ 深度合并，项目覆盖全局
项目资源 (cwd 向上搜索)
  ├── AGENTS.md / CLAUDE.md
  ├── .pi/SYSTEM.md / .pi/APPEND_SYSTEM.md
  ├── .pi/SKILL.md
  ├── .pi/PROMPT.md
  └── .pi/settings.json

策略：从全局到项目、从根目录到当前目录逐级加载。离代码越近的资源优先级越高。

解决的问题：同一个用户会在不同项目中工作。全局资源提供通用行为基线，项目资源提供团队约定、架构约束和局部技能。

不这样做会怎样：所有项目共享同一套行为规范，模型无法适配不同 repo；或者项目设置和全局设置冲突时行为不确定。

策略：不受信任项目不自动加载项目级 settings.json、AGENTS.md、SYSTEM.md 和技能资源。信任决策可以询问用户、记录、撤销。

解决的问题：防止恶意仓库通过项目说明文件植入 prompt injection，诱导 agent 忽略安全规则或泄漏信息。

不这样做会怎样：用户 clone 一个陌生仓库并打开，仓库中的指令就可能直接改写 agent 行为。上下文工程从第一步就被污染。

04

会话树：上下文不是线性聊天记录

pi 把会话存为树，而不是扁平消息数组。每次 LLM 响应、工具执行、压缩、模型变更、思维级别变更都是树上的 Entry。

线性聊天 vs 会话树结构Session

  class TA,TMAIN,TC,TD,TE,TNOTE section-1
从叶节点逆向重建上下文Session Tree
flowchart TB
  LEAF["从叶节点出发"] --> WALK["沿 parentId 向上追溯"]
  WALK --> CHECK{"当前节点类型？"}
  CHECK -->|"CompactionEntry"| STOP["停止追溯消息

  MODEL --> WALK
  TOOLS --> WALK
  STOP --> DONE["返回 AgentContext"]

策略：buildSessionContext() 从当前叶节点向上回放状态。遇到 CompactionEntry 就停止追溯早期原始消息，改用摘要；遇到模型、思维级别、工具状态变更 Entry，则恢复对应横向状态。

解决的问题：用户可以从历史任意节点 fork，压缩不会删除原始 Entry，会话恢复后也能保持上次选择的模型、工具和思维预算。

不这样做会怎样：线性历史无法无损探索。压缩等于永久删除消息；用户切换到高思维级别或禁用某工具后，恢复会话时状态又回到默认值。

深挖：会话树像产品里的“可回滚决策流”

对 AI 编码产品而言，用户经常需要尝试另一条路径：换一种重构方案、回退一个工具调用、从旧结论处重新提问。树形会话把这些探索变成低成本动作。被放弃的分支可以被摘要保存，主线不被污染，历史也不被丢弃。

05

系统提示词：最大的单次文本装配

buildSystemPrompt() 是上下文工程里最接近“拼接 prompt”的地方，但它不是简单拼接，而是按来源和语义组织成稳定结构。

策略：项目上下文用 XML 标签包裹，技能只放索引，不内联完整 SKILL.md。模型需要时再通过读取工具打开完整技能。

解决的问题：XML 标签帮助模型识别结构，也方便调试来源；技能按需加载避免系统提示词被数十个技能文件膨胀到不可控。

不这样做会怎样：20 个技能全部内联会在对话开始前消耗大量上下文窗口。模型注意力被技能说明稀释，真正的用户任务反而排在后面。

<available_skills>
  <skill>
    <name>pdf-generation</name>
    <description>Generates PDF reports from markdown.</description>
    <location>/path/to/skills/pdf-generation/SKILL.md</location>
  </skill>
</available_skills>
深挖：为什么技能只放索引

技能是能力目录，不是每轮都要读完的手册。把完整技能文件全部塞进系统提示词，会让模型在任务开始前就背负大量无关说明；只放名称、描述和位置，则保留了可发现性，也把是否读取完整内容交给当前任务需要。

06

Agent Loop：每轮调用前的最后装配线

进入 agent loop 后，上下文变化最频繁。模型每次调用前，消息都会经过 transformContext() 和 convertToLlm() 两道门。

transformContext()：剪枝发生在 AgentMessage 层

策略：在转换为模型协议前剪枝，因为 AgentMessage 比普通 role/message 携带更多类型信息，可以区分 bash 输出、自定义消息、压缩摘要、分支摘要。

解决的问题：剪枝时不能只看 user/assistant/tool。工具输出可能是当前任务的关键证据，UI 通知则不应该进入模型上下文。

convertToLlm()：最后一道门

AgentMessage 类型转换结果原因
User / Assistant / ToolResult原样传递LLM 原生支持。
BashExecutionMessage转换为 UserMessage模型不理解 bash 执行类型，需要翻译成可读上下文。
BranchSummaryEntry / CompactionEntry转换为带 XML 的 UserMessage摘要需要被后续模型感知。
内部 UI 或不可转换消息过滤掉只服务系统，不应消耗模型上下文。

excludeFromContext：用户静默执行

策略：用户用 !! 前缀执行 bash 时，输出保留在会话历史和 UI 中，但不进入 LLM 上下文。

解决的问题：用户需要“看一眼”的诊断输出，可能包含敏感信息或大量噪音，不一定希望模型看到。

深挖：两道门的产品分工

transformContext() 更像编排层，决定哪些 AgentMessage 还能代表当前任务；convertToLlm() 更像协议层，决定这些消息如何被翻译成模型能理解的 role 和 content。两者分开，才能同时保留产品语义和模型协议的稳定性。

07

Compaction：用摘要替代膨胀历史

上下文压缩不是“删掉旧消息”，而是把早期历史重写为结构化摘要。它在 agent 准备停止时触发，避免活跃工作中多插一次昂贵的 LLM 调用。

策略：默认开启压缩，保留最近约 20K tokens 原始消息，为响应预留约 16K tokens。压缩摘要固定为 Goal、Constraints、Progress、Key Decisions、Next Steps、Critical Context 六个 section。

深挖：好的压缩不是摘要文学

09

防欠给：让模型不必脑补

策略：系统提示词注入当前日期、cwd、可用工具、项目上下文和技能索引；压缩摘要强制结构化；会话恢复时还原模型、思维级别和活跃工具。

解决的问题：编码 agent 面对的是当前文件系统和当前项目约定，不是训练语料里的平均项目。缺少这些信息，模型只能猜。

不这样做会怎样：模型不知道工作目录，会把文件写到错误位置；不知道项目规则，会引入不符合风格的实现；不知道早期决策，会重复讨论或做出矛盾选择。

产品判断：防欠给不是把所有资料都塞进去，而是保证“当前决策所需的最低充分信息”稳定出现。

10

防错给：错误上下文比缺上下文更危险

策略：用 activeToolNames 控制工具暴露，用 SourceInfo 追踪 prompt 来源，用 TypeBox schema 验证工具参数，用确定性深度合并保证设置结果可复现。

来源分类：谁提供上下文，谁控制风险

上下文不只是“消息”。它来自开发者预设、用户提供和 LLM 自建三层，每层的稳定性、可控性和风险都不同。

第一层：开发者预设

策略：角色身份、能力边界、默认值和基础提示词保持稳定，通过 append prompt 和项目资源扩展。

产品含义：这是用户对 agent 建立预期的基础。它不应该频繁漂移。

第二层：用户提供

策略：项目设置、项目上下文、实时输入和工具结果分开管理。项目资源受信任边界保护；实时消息进入双队列；工具结果接受钩子和剪枝约束。

第三层：LLM 自建

策略：助手响应直接成为历史；压缩摘要使用固定模板；分支摘要跨分支传递上下文。

风险：LLM 的每一轮输出都会影响下一轮输入。如果早期偏差没有检查点，它会在上下文中自我强化。

深挖：来源分层解决的是责任边界

开发者预设负责稳定产品承诺，用户提供负责表达当前目标，LLM 自建负责沉淀执行过程。三者混在一起时，错误很难追责：到底是默认规则错了、用户输入变了，还是模型自己把偏差写进了历史。

深度洞察：上下文工程背后的产品架构

从 Pi 源码看，最有价值的部分，是那些看起来不是“上下文功能”，但实际上决定上下文是否可长期运行的架构机制。

Type
声明合并消息类型可扩展，核心包不变成中央注册表。
Event
事件系统上下文变化有审计轨迹，不靠猜。
Queue
双队列区分“立即转向”和“做完再处理”。
15

事件和钩子：可观测性不是事后补丁

策略：agent loop 和 harness 在关键节点发射结构化事件：请求前、payload 前、响应后、工具执行前后、session compact 前后、模型更新、工具更新等。

不这样做会怎样：只能看最终输出，无法判断是系统 prompt 漏了、工具列表错了、消息顺序乱了，还是插件改坏了 payload。

永不抛出的钩子合约

策略：钩子和工具异常不能摧毁代理循环，而是被转成错误工具结果或标准化事件错误，让模型和上层系统继续处理。

产品判断：在 AI Agent 中，上下文就是状态本身。崩溃不是简单重启进程，而是丢失推理链和用户信任。

深挖：事件系统为什么属于上下文工程

事件不只是日志。它记录的是上下文何时被装配、修改、拦截和失败。没有事件，产品只能解释最终回答；有了事件，才能定位是哪一层上下文让模型走向了错误路径。

16

双队列：用户插话的产品语义

用户在 agent 工作中输入新内容，可能有两种完全不同的意图：马上纠偏，或者当前任务结束后追加一个新任务。pi 用 steering 和 follow-up 两条队列表达这个差异。

不这样做会怎样：线性会话只能继续或删除。实验性工具调用一旦失败，就留在上下文里持续干扰后续模型。

18

韧性：失败也要进入上下文协议

生产级 agent 不可能假设 API、网络、工具、插件永远成功。pi 的选择是把失败纳入上下文协议，而不是让失败撕裂上下文。

机制策略产品价值
工具执行异常转换为 isError: true 的工具结果。模型知道工具失败，可以重试或换方案。
重试机制应用层分类错误，指数退避，成功后清零。临时网络错误不终止任务，上下文溢出不盲目重试。
StreamFn 合约传输失败序列化到流数据，而不是抛出。避免流式消息半截中断后留下不可解释状态。
设置迁移启动时幂等迁移旧设置和会话文件。升级后历史上下文和用户配置不丢失。

关键洞察：AI Agent 的状态不在某个数据库字段里，而在上下文、事件、工具结果和会话树的组合里。韧性设计本质上是在保护这组状态的连续性。

19

默认值：产品选择藏在配置里

pi 的默认值体现了一个明确倾向：对新用户来说，默认应该是不容易出问题，而不是理论上性能最优。

配置项默认值设计考量
toolExecutionparallel大多数读文件、搜索、列目录可并行，减少等待。
thinkingLevelmedium平衡推理质量与 token 消耗。
steeringModeone-at-a-time逐条注入，防止多条插话同时涌入。
followUpModeone-at-a-time让模型每处理一条后再看下一条。
compaction.enabledtrue默认防止长会话溢出。
compaction.reserveTokens16384为模型响应预留空间。
compaction.keepRecentTokens20000保留最近工作区的原始上下文。
retry.maxRetries3覆盖多数 API 波动，避免无限重试。
20

总结：面向生产级 AI Agent 的设计原则

pi 的上下文工程不是单点优化，而是一套产品系统：资源发现、信任边界、会话树、压缩、防错、事件、队列和韧性机制共同工作。

  1. 上下文即状态。对 AI Agent 来说，历史、工具结果、摘要和配置状态共同定义当前世界。保护上下文，就是保护产品连续性。
  2. 上下文预算要主动管理。不要把选择权完全交给模型。哪些工具可见、哪些技能可读、哪些消息进入上下文，都应由编排层管理。
  3. 错误信息要显式进入协议。工具失败、钩子异常、流式中断都要被结构化表达，而不是让循环崩溃。
  4. 用户意图需要时机语义。同一句用户输入，在工作中途和工作结束后含义不同。steering 和 follow-up 的差异是产品语义，不只是队列实现。
  5. 默认值是一种产品承诺。新用户不调参也应该得到稳定行为，高级用户再通过钩子和配置覆盖。

最终判断：pi 把 AI Agent 当成一个需要长期运行、可观察、可恢复、可扩展的产品系统，而不是一个简单的模型调用脚本。这正是上下文工程从 prompt 技巧升级为产品架构的分界线。
